# Lesson 5: Jotai 状态建模与渲染层交互闭环

## 本课目标
- 用 Jotai 建立 NoteMark 的渲染层状态模型
- 串起列表选中、内容编辑、新建笔记、自动保存等关键交互
- 让组件只关心显示，状态与动作集中管理

## 对应 practice 主线
这一课对应 practice 中的 Jotai 接入、笔记状态组织、保存动作和界面联动部分。

## 当前仓库映射
- `src/renderer/src/store`：原子状态和派生动作
- `src/renderer/src/hooks`：保存、选中、初始化等复用逻辑
- `src/renderer/src/components`：消费 atom 的 UI 组件

## 学习重点
- 原子状态与派生状态的区别
- 为什么不要把保存逻辑散落在多个组件里
- 自动保存为什么必须有节流、去抖或提交时机控制

## 推荐状态拆分
- `notesAtom`：全部笔记元数据或当前已载入的笔记集合
- `activeNoteIdAtom`：当前选中笔记
- `activeNoteAtom`：派生得到的活动笔记
- `editorDraftAtom`：当前编辑中的临时内容
- `saveNoteAtom`：统一的保存动作

## 可执行实践路线
1. 先完成状态图
   - 列出哪些状态属于全局
   - 列出哪些状态只属于组件局部
2. 建立最小 atom 集合
   - 笔记集合
   - 当前活动笔记
   - 编辑草稿
   - 保存动作
3. 串起列表与编辑器
   - 左侧点击切换活动笔记
   - 右侧标题和正文实时更新草稿状态
4. 加入自动保存策略
   - 草稿变化后触发保存动作
   - 保存后更新列表中的摘要和更新时间
5. 处理关键边界
   - 新建后立即成为活动笔记
   - 删除当前笔记后自动选中下一条或进入空状态
   - 切换过快时不丢失最近一次编辑

## 本课产出
- 一套清晰的 Jotai 状态模型
- 一个统一保存入口
- 一份渲染层交互时序图

## 完成标准
- 列表、标题、编辑器共享同一套可信状态
- 自动保存不会产生明显抖动或重复写入
- 新建、切换、删除三类动作都能闭环

## 功能规划结果
渲染层状态只负责两类事情：
- 表达 UI 当前状态
- 发起对 preload API 的调用

以下内容不要放在 renderer 里直接做：
- Node 文件系统读写
- 文件路径拼接
- 窗口原生控制实现

## 建议进阶练习
- 记录 `dirty` 状态并在保存完成后清零
- 增加“保存中”与“保存失败”提示
- 将选择逻辑与保存逻辑拆为独立 hooks
